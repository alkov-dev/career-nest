# 🚀 План разработки проекта Career

**AI-платформа для подбора кандидатов и вакансий**

---

## 📊 Общая информация

- **Стек:** NestJS + Next.js 15 + PostgreSQL + pgvector + Redis + BullMQ + MinIO + OpenAI
- **Миграции:** Goose (SQL-native)
- **ORM:** Prisma
- **AI:** GPT-4o, text-embedding-3-large
- **Email:** Resend
- **Хранилище:** MinIO (S3-совместимое)

---

## ️ Архитектура базы данных

**27 таблиц** в 6 блоках:

1. **Core** — users, candidate_profiles, employer_profiles, companies
2. **Skills & Languages** — skills, skill_aliases, languages, language_aliases, candidate_skills, profile_languages
3. **Jobs** — jobs, job_skills, job_languages
4. **Matching** — applications, match_scores, saved_jobs, job_history, matching_weights, geo_clusters
5. **CV & AI** — cv_uploads, experiences, cv_analyses, cv_resume_insights, cover_letters
6. **RAG & Communication** — kb_chunks, notifications, messages

## 1.4. JWT авторизация на куках

### Что делаем:
- Создаём модуль Auth
- Реализуем регистрацию (candidate/employer/admin)
- Реализуем логин с проверкой пароля (bcrypt)
- Генерируем JWT токены (access + refresh)
- Сохраняем токены в HttpOnly cookies (не в localStorage)
- Создаём `JwtAuthGuard` для защиты эндпоинтов
- Создаём `RolesGuard` для проверки ролей
- Реализуем логаут (очистка cookies)
- Реализуем refresh token rotation
- Создаём эндпоинт `/auth/me` для получения текущего пользователя

### Эндпоинты:
| Метод | Путь | Описание | Роль |
|-------|------|----------|------|
| POST | `/auth/register` | Регистрация (создание user + profile) | all |
| POST | `/auth/login` | Логин (возвращает cookies) | all |
| POST | `/auth/logout` | Логаут (очистка cookies) | all |
| POST | `/auth/refresh` | Обновление access token | all |
| GET | `/auth/me` | Текущий пользователь | authenticated |

### Структура модуля Auth:


### Как работают cookies:
- **Access token** → HttpOnly cookie, `maxAge: 15 минут`
- **Refresh token** → HttpOnly cookie, `maxAge: 7 дней`
- При запросе → Guard читает access token из cookie
- При истечении access → фронтенд вызывает `/auth/refresh`
- При logout → оба cookie удаляются, refresh token инвалидируется в БД

### Критерий готовности:
✅ Можно зарегистрироваться, залогиниться, получить защищённый эндпоинт с куками

## 1.6. Генерация embeddings через BullMQ

### Что делаем:
- Настраиваем BullMQ + Redis
- Создаём очередь `embeddings` для генерации векторов
- Создаём worker для обработки задач из очереди
- Реализуем интеграцию с OpenAI API для генерации embeddings
- Создаём сервис для обновления векторных полей через raw SQL
- Добавляем задачи в очередь при создании/обновлении:
  - Навыков (`skills.embedding`)
  - Профилей кандидатов (`candidate_profiles.summary_embedding`, `location_embedding`)
  - Вакансий (`jobs.embedding`, `location_embedding`)
- Настраиваем retry policy и error handling
- Добавляем логирование использования токенов OpenAI

### Структура модуля очереди:
src/queue/
├── queue.module.ts
├── queue.service.ts # Добавление задач в очередь
└── processors/
└── embeddings.processor.ts # Worker для обработки задач



### Типы задач в очереди:
| Тип задачи | Описание | Триггер |
|------------|----------|---------|
| `generate.skill.embedding` | Генерация embedding навыка | Создание/обновление навыка |
| `generate.profile.summary` | Генерация embedding summary | Обновление summary профиля |
| `generate.profile.location` | Генерация embedding локации | Обновление города профиля |
| `generate.job.embedding` | Генерация embedding вакансии | Создание/обновление вакансии |
| `generate.job.location` | Генерация embedding локации вакансии | Обновление location вакансии |

### Настройки очереди:
- **Retry attempts:** 3
- **Backoff:** exponential (1s, 2s, 4s)
- **Concurrency:** 5 (не перегружать OpenAI API)
- **Rate limit:** 10 задач в минуту
- **Dead letter queue:** для_failed задач

### Критерий готовности:
✅ При создании навыка/профиля/вакансии автоматически генерируется embedding через очередь

# 🎯 Критические риски и решения

## Риск 1: OpenAI API медленный и дорогой
**Решение:** Все AI-операции через BullMQ очереди, кэширование результатов, использование более дешёвых моделей (gpt-4o-mini) где возможно.

## Риск 2: Векторный поиск медленный на больших данных
**Решение:** HNSW индексы, ограничение количества кандидатов для детального анализа (топ-50), пагинация.

## Риск 3: Платежи не работают
**Решение:** Тестирование в sandbox с самого начала, проверка webhook'ов, обработка отказов.

## Риск 4: Система не масштабируется
**Решение:** Load testing на этапе 9, оптимизация запросов, кеширование, горизонтальное масштабирование.

## Риск 5: Баги в production
**Решение:** Sentry для мониторинга, логирование, health checks, возможность быстрого отката.
